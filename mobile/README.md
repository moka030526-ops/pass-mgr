# pass-mgr mobile — Android & iOS

Native **Android** and **iOS** apps for the pass-mgr estate vault, built as a
single **Compose Multiplatform** (Kotlin) UI on top of the **same audited Rust
core** as the desktop app. No crypto, storage format, or recovery logic is
reimplemented — the mobile apps call the exact `pass-mgr-core` code through a
thin [UniFFI](https://mozilla.github.io/uniffi-rs/) boundary, with the Kotlin
bindings generated by [Gobley](https://gobley.dev).

> **Status — v1 is a read-only viewer.** Unlock with your two passwords, browse
> the five tabs, view an entry, reveal/copy a password. Creating and editing
> entries, documents, backup, and changing passwords are deliberately **not** in
> v1 (do those on the desktop app for now). See [Roadmap](#roadmap).

## Architecture

```
            ┌──────────────────────────────┐   ┌──────────────────────────────┐
            │  Android app (Jetpack/ART)    │   │  iOS app (UIKit host)         │
            │  MainActivity → App()         │   │  iOSApp → MainViewController  │
            └──────────────┬───────────────┘   └───────────────┬──────────────┘
                           │   one shared Compose Multiplatform UI (Kotlin)     │
                           └────────────────────┬───────────────────────────────┘
                                                 │  com.passmgr.ffi.* (generated)
                          Gobley + UniFFI bindings (Kotlin)
                            • Android: JNI/JNA → libpass_mgr_ffi.so (per ABI)
                            • iOS:     cinterop → libpass_mgr_ffi.a (XCFramework)
                                                 │
                           ┌─────────────────────┴─────────────────────┐
                           │  pass-mgr-ffi   (this repo: crates/)        │
                           │  opaque Vault handle + flat DTOs + errors   │
                           └─────────────────────┬─────────────────────┘
                           ┌─────────────────────┴─────────────────────┐
                           │  pass-mgr-core  (AUDITED, reused verbatim)  │
                           │  Argon2id ×2 · XChaCha20-Poly1305 ·         │
                           │  partitioned crash-safe volume/manifest ·   │
                           │  fails-closed recovery                      │
                           └─────────────────────────────────────────────┘
```

The repository is a Cargo workspace:

| Crate | Role |
|-------|------|
| `crates/pass-mgr-core` | Headless, audited vault core. No UI/OS deps. `#![forbid(unsafe_code)]`. Reused by every front-end. |
| `crates/pass-mgr-ffi`  | Thin UniFFI wrapper (opaque `Vault`, flat DTOs, no-leak error mapping). The only crate that allows `unsafe` (the UniFFI scaffolding). |
| `crates/pass-mgr-desktop` | The existing CLI + ratatui TUI + egui GUI binaries (`pass-mgr`, `pass-mgr-gui`). |
| `mobile/`              | This Compose Multiplatform Gradle project. Points Gobley at `crates/pass-mgr-ffi`. |

The mobile UI lives entirely in Kotlin (`mobile/composeApp/src/commonMain`); the
Rust core stays headless.

## Building Android (works on Linux/macOS/Windows)

### 1. One-time toolchain install

The repo ships a no-sudo, userspace installer that fetches a JDK, the Android
SDK + NDK, the Rust Android targets, and cargo-ndk under `~/toolchains`:

```bash
bash mobile/scripts/install-android-toolchain.sh
source ~/toolchains/env.sh        # JAVA_HOME, ANDROID_HOME, ANDROID_NDK_HOME, PATH
```

(Or use your existing Android Studio / SDK + NDK and JDK 17.)

### 2. Build the APK

```bash
source ~/toolchains/env.sh
mobile/gradlew -p mobile :composeApp:assembleDebug
# -> mobile/composeApp/build/outputs/apk/debug/composeApp-debug.apk
```

Gobley cross-compiles `libpass_mgr_ffi.so` for each ABI (`arm64-v8a`, `x86_64`),
generates the Kotlin bindings, and packs everything into the APK.

Install on a device/emulator: `adb install -r <the apk>`.

> The pinned NDK is `30.0.14904198` (see `composeApp/build.gradle.kts`
> `ndkVersion`). If you use a different NDK, change that line.

## Building iOS (requires a Mac + Xcode)

**iOS cannot be built or signed on Linux** — Apple's toolchain is macOS-only.
On a Mac with Xcode + the Rust iOS targets:

```bash
rustup target add aarch64-apple-ios aarch64-apple-ios-sim x86_64-apple-ios
```

The iOS Compose targets are already declared in `composeApp/build.gradle.kts`
(guarded by `GobleyHost.Platform.MacOS.isCurrent`, so they're inert elsewhere).
Gobley builds the Rust **staticlib** and links it via Kotlin/Native cinterop.

The Kotlin iOS entry point is `composeApp/src/iosMain/kotlin/com/passmgr/MainViewController.kt`
and the Swift host lives in `mobile/iosApp/`. Open `mobile/iosApp` in Xcode (or
generate the project with the KMP/Compose tooling), point it at the `ComposeApp`
framework Gradle produces, and run. See `mobile/iosApp/README.md`.

## Getting your vault onto the phone (offline by design)

pass-mgr **never syncs** — there is no cloud, no network permission, no account.
To view your vault on the phone you move the encrypted files yourself:

1. On desktop, make a backup (`pass-mgr backup <vault> <dir>`) — it copies the
   encrypted `vault.pmv` + `manifest/` + `volume/` folder.
2. Copy that folder into the app's private storage:
   - **Android:** `adb push <vault-folder>/. /data/data/com.passmgr/files/vault/`
     (debug builds; on a real device use the in-app import — *roadmap*).
   - **iOS:** drop it into the app's Documents via Finder/Xcode, or the in-app
     import — *roadmap*.
3. Unlock with the same two passwords.

The encrypted format is **identical** across desktop and mobile, so a vault made
on either opens on the other with no conversion.

## Security notes (disclosed downgrades vs desktop)

Running in a mobile sandbox is weaker than the desktop in specific, documented
ways. None of these weaken the cryptography; they are about where cleartext can
briefly live and OS guarantees the desktop has that phones don't:

- **Cleartext crosses the FFI.** Viewing or copying a field returns it as a
  Kotlin/Swift string into the managed heap, where `ZeroizeOnDrop` can't reach
  and the GC/ARC may duplicate it. The app minimizes copies and fetches edit
  history (which can embed old/new passwords) only on demand.
- **No swap-locking (`mlock`).** The mobile build compiles with the `mlock`
  feature **off** (phones grant apps no `mlock` budget), so the derived key isn't
  pinned out of swap — `region` isn't even linked. Mitigated by: Android has no
  swap by default, iOS encrypts its swap, and the key is still zeroized on drop.
- **No cross-process single-writer lock.** The desktop's `pass-mgr.lock` is
  dropped; a single app process doesn't need it, and the FFI serializes all calls
  behind one mutex.
- **Clipboard auto-clear is best-effort.** The app re-implements the desktop's
  15-second clipboard wipe in Compose; the OS gives no auto-clear contract.
- **Keep the vault in app-private storage only** (the app does). On SAF / iCloud-
  backed trees the POSIX rename + dir-fsync atomicity the crash-safety relies on
  is weaker, and a silent cloud restore of an old snapshot would defeat the
  generation-based rollback signal — which is why the app sets `allowBackup=false`
  / excludes the vault from cloud backup.

## Roadmap (post-v1, additive)

Editing (`upsert`/`delete`), the password generator, category dropdowns, theme,
biometric/Keychain/Keystore unlock gating, in-app encrypted import/export via the
document picker, and then documents (attach/export), rekey, backup, and
compaction. The `mlock`/single-writer-lock feature gates are already in place (the
mobile build compiles without either); the main remaining core change is a
byte-buffer document API for attach/export from the platform document picker.
