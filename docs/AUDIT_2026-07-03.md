# Deep Audit — 2026-07-03

A multi-agent, adversarially-verified bug hunt and security review across the **entire**
workspace (core, desktop GUI/TUI/CLI, ffi, fuzz), triggered by the new asset↔account link
feature but deliberately scoped to the whole codebase. Ten independent finder lenses
produced candidate findings; each survivor was confirmed by **≥2 independent verifier
agents** (with a tiebreaker agent on split votes) under the stated threat model before
being acted on. All six libfuzzer targets were also re-run (~128 M executions total, **zero
crashes**), and `cargo audit` / `cargo deny` were brought back to green.

**Threat model (security):** a standalone, OFFLINE two-password encrypted estate vault. In
scope: theft of / tampering with the on-disk vault files (confidentiality + rollback
detection), a crafted vault / **hostile merge-source vault** / document fed to
open/merge/import/extract, malicious paths/filenames/record content, a local multi-user
filesystem (permissions, symlinks, TOCTOU), and secrets lingering in memory/disk/terminal.
Out of scope: network attacks (the app opens no sockets), and an attacker already running as
the user with the vault unlocked in RAM.

**Result:** no Critical findings. Two **High** and several **Medium** findings were confirmed
and **fixed**; the High ones were *availability/data-loss* bricks (a write path that could
produce a vault its own reader rejects), not confidentiality breaks. The core cryptography,
atomic-write discipline, and input sanitization from the prior audit rounds held up. A few
low-severity residuals are documented with rationale at the end.

## Findings and fixes

### A-1 / A-6 — Manifest entry-count cap enforced only on read → brick (High via merge)

`MAX_MANIFEST_ENTRIES` (100 000) is a fail-closed DoS guard that `load_manifest`
(`storage.rs`) rejects a larger manifest with (`TooLarge`), and `open` does **not** treat
`TooLarge` as rebuildable. But the **write** path `VolumeStore::put` grew the active
partition's manifest one entry per new document with no matching cap: `target_partition`
rolled to a fresh partition only on the **byte** cap, so under the default 256 MiB partition
size a partition admitted ~230 k tiny frames — well past 100 k — before rolling. The result:
adding ~100 k small documents (A-1, normal use) **or** a hostile merge source packing many
tiny blobs into one destination partition (A-6, `apply_merge_from` → `put`) commits fine,
then the **next open permanently bricks** an otherwise-intact vault with no in-app recovery —
the worst outcome for an estate vault.

**Fix:** `target_partition` now rolls to a fresh partition when **either** the byte cap **or**
the entry-count cap would be exceeded, so every partition manifest stays ≤ the read-side cap
and the write/read paths agree. Updates to an existing id keep their partition (retain+push
holds the count constant). One fix closes both the normal-use and the hostile-merge vector.
Test: `put_rolls_to_a_new_partition_at_the_manifest_entry_cap`.

### A-7 — Vault file could exceed the read-side `MAX_VAULT_SIZE` → unopenable

The same "write path doesn't enforce what the read path rejects" class, for `vault.pmv`.
`write_vault_file` serialized and atomically wrote the whole vault with no size check, while
every reopen goes through `read_capped_vault`, which rejects a file larger than
`MAX_VAULT_SIZE` (256 MiB). A large additive merge (records copied verbatim with full
history) could push the file over and silently brick the destination.

**Fix:** `write_vault_file` now fails **closed** (`VaultError::TooLarge`) before touching the
disk when `header + ciphertext` would exceed `MAX_VAULT_SIZE`, leaving the previous,
still-openable generation intact and surfacing the error to the caller.

### A-2 — Cross-epoch recovery silently rolled back a committed password change

`decrypt_with_redundancy` derives a key per **candidate salt** and decodes each redundant
copy under its own-salt key. So an old-epoch `bak` (pre-rekey salt) still decodes when the
user supplies the **OLD** passwords — falsifying the documented F3 assumption that "recovery
only ever uses a copy that decodes under the current salt." Scenario: redundancy on;
`change_password` commits (new salt); the best-effort cleanup fails on a transient EIO/EACCES
and strands an old-epoch `bak`; the next open with the OLD password fails on the new-salt
primary, then recovery decodes the stranded old `bak` and "succeeds" with the **pre-rekey**
vault — re-validating the old (possibly compromised) passwords, and letting the writable-open
sweep/heal **destroy the new-epoch copies** and overwrite `vault.pmv` under the old key. A
silent password-change rollback and forward-secrecy loss.

**Fix:** recovery is now confined to copies sharing the primary's salt — **but only when that
salt is corroborated by at least one redundant copy**. This precisely separates the two cases
that look identical from a header alone: (a) *wrong password on an intact primary* (the
rollback vector — the current-epoch mirror/bak1 corroborate the salt, so the old-epoch `bak`
is excluded and the old password fails closed), versus (b) *bit-rot inside the primary's salt
bytes* (the garbage salt matches no copy, so the restriction is **not** applied and the intact
mirror under a different, correct salt stays usable). Tests:
`old_password_cannot_roll_back_a_committed_rekey_via_a_stranded_bak` (new) and
`recovers_from_mirror_when_primary_salt_corrupt` (regression, still green).

### A-3 — Ring rotation ingested an unvalidated primary

`save_internal` read the outgoing generation with `read_capped_vault` (bytes only, no AEAD
check) and `rotate_generations` ringed it into `bak1` while deleting the oldest good
generation. A primary that bit-rotted on disk during the session — or a corrupt primary left
by a heal whose best-effort re-save failed — would be ingested as `bak1`, **replacing a
recoverable snapshot with an unrecoverable one**.

**Fix:** the outgoing generation is ringed in only if it **decodes under the current key**;
otherwise it is dropped and the ring is pruned only, preserving the good generations. One
extra (fast, symmetric) AEAD decrypt per save when redundancy is enabled.

### A-8 — Same-frame keyboard nav retargeted deferred destructive actions (GUI)

In the Accounts tab the deferred phase applied `select = select.or(nav_target)` — swapping
`edit_account` to the arrow-key target — **before** the deferred `FormAction::Delete` /
confirm handlers, which then operated on the just-swapped buffer. Because egui batches input
events per repaint, a keyboard arrow can land in the **same frame** as a Delete/confirm
button click, so the destructive action captured against the shown record could execute
against the **neighboring** one. (The linked-accounts review had already fixed the narrow
delete-*confirm* sub-case with a confirm-time id re-check; this is the general fix.)

**Fix:** the keyboard nav (`nav_target`) is now **suppressed** in any frame that also carries
a record-targeted action (New / Delete / confirm / cancel / Generate). A pointer click on a
list row (`select`) cannot co-occur with another button click, so it is still honored.

### A-9 — Per-vault GUI state survived lock→re-unlock (secret leak across vaults)

Two error paths drop the open vault back to the unlock screen **without** going through the
constructor — a change-password failure and a **merge-apply failure (reachable via a hostile
merge-source vault, in scope)**. Neither cleared the seven `edit_*` buffers (which hold
cleartext passwords), the armed `pending_account_delete`, the filters, or the reveal toggles,
and opening the next vault installed it without resetting them. So vault A's record — and its
cleartext secret in the edit buffer — could be displayed in, and saved into, a different
vault B.

**Fix:** a new `reset_per_vault_ui_state()` clears every per-vault buffer/filter/reveal/armed
state to its fresh-launch default, called on **every** successful open before the new vault is
installed. The edit buffers are `Zeroize`-on-drop, so replacing them also wipes any secret
they held. Test: `opening_a_vault_clears_prior_per_vault_ui_state`.

### A-10 — Delete-rollback re-inserted the dirty edit buffer (GUI)

`delete_current` built each persist-failure rollback closure from `self.edit_*.take()` — the
live edit buffer the form mutates directly — not from the record `records::remove` actually
deleted. When the user had typed since loading (e.g. a new password) and the delete's persist
then failed, the rollback re-inserted the **buffer's unsaved edits**, which a later successful
save would silently commit with no history diff — a delete the user was told *failed*
silently changing a saved value.

**Fix:** every tab's rollback now snapshots the **stored** record (its last-saved state) before
`remove` and re-inserts *that* on failure, while still restoring the user's editing session
(`edit_*`) for continuity. Applied to all seven record tabs. Test (fault-injected):
`delete_rollback_restores_the_saved_record_not_the_dirty_edit_buffer`.

### A-5 — Partial-plaintext shred warning under-reported cleartext in pre-existing subtrees

`warn_partial_plaintext` (used when a plaintext `export-tree`/`extract` fails mid-walk) diffed
only **top-level** entry names against a pre-run snapshot and returned silently when no *new*
top-level entry appeared. But an export into a directory that already contained a same-named
entry (e.g. a `documents/` dir from a prior export) writes fresh cleartext **inside** that
pre-existing subtree — invisible to a top-level diff — so the user could be told nothing
landed while decrypted secrets were written.

**Fix:** when the output directory was **non-empty** before the run, the warning now
conservatively flags that cleartext may have been written into the pre-existing directories
too and to treat the entire directory as potentially holding decrypted secrets.

## Supply chain (`cargo audit` / `cargo deny`)

`cargo audit` flagged five advisories; all resolved or documented:

- **`anyhow` 1.0.102 → 1.0.103** and **`memmap2` 0.9.10 → 0.9.11** — patch bumps applied
  (`cargo update`), clearing the two unsoundness advisories.
- **`quick-xml` 0.39.4** (two DoS advisories) — reachable **only** through
  `wayland-scanner`, a build-time proc-macro that parses the vendored Wayland protocol XML
  shipped inside the `wayland-*` crates; it never sees attacker-controlled input and never
  ships in the binaries. `wayland-scanner` pins `^0.39`, so the fixed 0.41 is unreachable
  until upstream bumps. Documented + ignored in `.cargo/audit.toml` and `deny.toml`.
- **`ttf-parser` 0.25.1** (unmaintained) — reaches us via egui's text stack and only parses
  the fonts egui **bundles** into the binary (trusted, shipped data; the app loads no
  user/system fonts). Documented + ignored in `deny.toml`.

`cargo deny check` and `cargo audit` are both **green** with those documented ignores.

## Residuals (documented, not fixed)

- **A-4 — Core-dump suppression.** A native crash can persist the whole decrypted vault
  (including the mlock'd `Key`, which is still captured in a dump) to disk. The clean in-process
  mitigations (`prctl(PR_SET_DUMPABLE, 0)`, `setrlimit(RLIMIT_CORE, 0)`) require raw libc calls,
  but the desktop crate is `#![forbid(unsafe_code)]` — a **load-bearing auditability property**
  (DESIGN.md req. 13) — and this offline app deliberately minimizes dependencies, so neither an
  `unsafe` block nor a new crate is a good trade for a defense-in-depth item. **Operational
  mitigation:** run under `ulimit -c 0` (shell) or `LimitCORE=0` (systemd), the standard way to
  disable core dumps for a sensitive process. Recorded in `docs/HARDENING.md`.
- **FFI `RekeyPending` correct-password oracle (Low).** Two `RekeyPending` returns in
  `open_inner` fire **after** a successful decrypt (a reader-vs-writer rekey race), so a
  local dir-write attacker who plants a persistent `.rekey` dir + corrupts the volume can
  distinguish a correct password (`RekeyPending`) from a wrong one (`WrongPasswordOrCorrupt`)
  on the FFI surface — the same oracle class as the fixed audit R-3, via a different variant.
  **Not fixed** because the only correct fix (folding `RekeyPending` into the generic error on
  the FFI open path) also suppresses the **legitimate** pre-decrypt rekey-pending signal on
  mobile, and the oracle only leaks password *correctness* to an attacker who already has
  vault-directory write access and is guessing online against the app — low practical value.
  Flagged for a maintainer decision.
- **ASCII-only category case-dedup (Low).** Type-list dedup/removal use
  `eq_ignore_ascii_case`, so non-ASCII case variants (e.g. `Café`/`CAFÉ`) can coexist as
  duplicate categories. Consistent with every other matcher in the codebase; the spoofy-char
  sanitizer strips zero-width/bidi but not accented duplicates. Cosmetic; a Unicode
  case-folding change would touch every usage helper and is deferred.
- **`account_tree` blank-title leaf (Low).** Unlike `asset_tree`, the account grouping leaf
  has no blank fallback, so a whitespace-only title (e.g. from a verbatim merge) renders a
  blank leaf. Cosmetic.
- **Test gaps (Low).** The `import_tree` `is_safe_blob_id` traversal guard and the
  deleted-secret export-tombstone filter are correct by inspection but lack dedicated
  reachability tests, and the `merge_from` fuzzer builds only benign sources (so the untrusted
  merge blob-id/path validation is not fuzzed). Recommended as follow-up test additions.
